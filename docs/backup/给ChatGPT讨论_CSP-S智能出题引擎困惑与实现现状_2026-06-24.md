生成时间：2026-06-24

适用项目目录：`D:\lyh\agent\agent-frame\newLuogu`

本文档目的：把当前项目里“已经实现的东西”“真正有价值但还不足的地方”“用户当前困惑”“接下来该如何把它变成 60 天训练决策系统”完整摊开，方便继续和 ChatGPT 深度讨论。

本文档不是宣传稿，也不是验收报告。它更像一份问题说明书：当前系统已经能跑，但离“真正准确地分析学生、并给出 60 天冲刺一等的训练方案”还有明显距离。

## 1. 用户真正关心的问题

用户的核心诉求不是简单的“把大模型接进去”，也不是“生成几个题包证明功能能跑”。

用户真正关心的是：

1. 分析是否准确。
2. 能不能真正找出学生冲 CSP-S 一等奖的关键短板。
3. 能不能把短板转成 60 天内可执行、可验证、可动态调整的训练方案。
4. 中间过程要怎么做，怎么每天训练，怎么每周复盘，怎么判断有没有变强。
5. 大模型生成题到底能不能服务于训练，而不是变成形式上的 AI 出题。

用户的困惑可以概括成一句话：

> 这个系统现在看起来能诊断、能出题、能生成任务包，但它到底是否真的“分析准了”？如果分析还不够准，下一步应该补什么，才能支撑一个 60 天达成目标的训练闭环？

## 2. 当前项目目标重新定义

原需求文档的目标是“CSP-S 题目归类与智能出题引擎”，但从用户最新表达看，目标应该升级为：

> 基于学生历史提交、错题、代码特征、题目簇和大模型诊断，构建一个能够持续识别薄弱点、生成针对性训练任务、安排 60 天冲刺计划、并通过训练结果反向修正诊断的闭环系统。

所以这个项目不应该只被看成“出题工具”，而应该被看成四层系统：

1. 分析层：从历史数据中识别学生能力画像。
2. 诊断层：把题目簇、错误类型、算法模块转成真实薄弱点。
3. 训练层：把薄弱点转成选题、变式题、原创诊断题、每日任务包。
4. 闭环层：把学生后续训练结果回填，再重新诊断和调整计划。

当前已经实现了前 3 层的雏形，但第 4 层还明显不足。

### 4.1 对学生画像的当前解释

当前诊断结果指向一个比较明确的画像：

1. 学生不是完全缺基础题能力。
2. 最大问题集中在“难题拆解能力”和“模型识别能力”。
3. 数学、部分分、区间 DP 是前 20 天最应该抓的主线。
4. 区间 DP 和普通 DP 的问题不是单点知识没学，而是状态定义、初始化、枚举顺序、边界验证这一整套流程不稳定。
5. 图论、二分、并查集也有明显建模问题，但优先级应排在前三大弱点之后。
6. 如果目标是 CSP-S 一等奖，训练不能只按题单刷题，而必须训练“考场拿分策略”。

### 4.2 最关键判断

当前系统给出的最关键判断是：

> 学生当前最需要提升的不是刷题数量，而是看到难题后如何拆分 30/50/70 分，如何识别模型，如何稳定写出状态、转移、初始化、边界和复杂度。

所以 60 天计划应该以“部分分策略 + 模型识别 + 稳定实现”为主，而不是简单堆题量。

## 5. 当前已经实现了什么

### 5.1 大模型簇诊断链路

已实现能力：

- 可以读取题目簇 evidence pack。
- 可以调用 DeepSeek 对问题簇做诊断。
- 可以生成 `cluster_llm_diagnosis.json`。
- 可以记录 trace 到 `cluster_llm_diagnosis_traces.json`。
- 可以控制 `DIAGNOSIS_LLM_LIMIT`，避免一次性全量调用过慢。
- 可以用 `DIAGNOSIS_LLM_REQUIRE_REAL=true` 要求真实调用，不允许无声兜底。
- 可以把簇诊断传播到题目级诊断。
- 可以重新生成弱点报告和训练建议。

相关实现文件：

- `scripts/diagnosis/deepseek-diagnosis-client.ts`
- `scripts/diagnosis/diagnosis-lib.ts`

已修复的关键问题：

- 原来如果大模型返回内容里有非法 JSON 转义，会解析失败。
- 原来只读取 `choices.0.message.content`，没有兼容 `reasoning_content`。
- 原来报告总览文案固定写“规则全量覆盖”，现在改为动态显示真实 LLM 覆盖数量。
- 现在 JSON 提取器会扫描第一个真正合法的 JSON 对象，避免推理文本中的 `{k-1}` 等内容被误判为 JSON 起点。

当前不足：

- 只跑了 5 个真实大模型簇诊断，99 个仍然是规则兜底。
- 对“分析是否准确”的验证仍不充分，因为缺少学生本人训练后的反馈数据。
- evidence pack 可能压缩过度，大模型看到的信息不一定足够判断深层原因。

### 5.2 智能出题链路

已实现能力：

- `problem-bank:select`：根据弱点选历史题。
- `problem-bank:mutate`：生成 8 个变式题规格。
- `problem-bank:generate`：生成 3 套原创诊断题包。
- `problem-bank:pack`：生成 7 天训练任务包。
- `problem-bank:all`：串起完整题库和出题引擎。

相关实现文件：

- `scripts/problem-bank/problem-bank-lib.ts`

已修复的关键问题：

- 大模型返回路径读取问题。
- 大模型 JSON 非法转义问题。
- 大模型返回字段名不稳定问题，例如 `statement`、`statementMd`、`stdCpp` 等。
- 大模型题包缺字段时会二次修复调用。
- `problem-bank:pack` 曾经会把 LLM 题包覆盖成规则兜底题包，现在已修。
- `generationSource=LLM` 时 status 会归一化为 `LLM_GENERATED_NEEDS_REVIEW`。

当前不足：

- 生成题包只是“需要审题”的训练草稿，还不能直接当正式比赛题。
- 没有自动编译 `bruteforce.cpp`、`std.cpp`、`generator.cpp`。
- 没有自动对拍验证。
- 没有自动生成完整测试数据。
- 没有自动判定题目是否有歧义、是否样例正确、是否复杂度匹配。
- 生成题和真实洛谷题之间的难度对齐还不够精确。

### 5.3 7 天任务包

当前已生成：

- `data/problem-bank/task_pack_7d.md`

当前任务包结构：

- Day 1：T1 稳定性 + 边界检查
- Day 2：数学建模 + 部分分拆解
- Day 3：部分分策略
- Day 4：区间 DP
- Day 5：普通 DP
- Day 6：图论建模
- Day 7：综合验证 + 原创诊断题

当前不足：

- 只有 7 天任务包，不是正式 60 天计划。
- 每天任务不够细，没有拆成“读题、建模、编码、提交、复盘、回填”的标准动作。
- 没有周目标、阶段目标和退出标准。
- 没有训练日志数据结构。
- 没有根据每日结果动态调整后续任务。

## 6. 当前实现不足的地方，按严重程度排序

### P0：分析准确性还不够

这是最核心的问题。

当前系统已经能给出看起来合理的薄弱点，但还不能证明它足够准确。原因：

1. 大模型只诊断了 5 个簇。
2. 大多数簇还是规则诊断。
3. 没有学生对诊断结果的人工反馈。
4. 没有后续训练表现来验证诊断是否命中。
5. 没有把“错因”细化到可操作层面，例如：
   - 是读题误解？
   - 是模型没识别？
   - 是知道模型但不会转移？
   - 是边界没处理？
   - 是代码实现不稳定？
   - 是考场时间分配错误？

当前诊断能告诉我们“数学建模弱”“区间 DP 弱”，但还没有足够精细地回答：

> 到底是哪个环节导致失分？

### P0：60 天计划还没有真正产品化

现在有 7 天任务包和一个粗略 60 天思路，但还没有正式计划文档。

正式 60 天计划应该包含：

1. 每周目标。
2. 每日任务结构。
3. 每日训练量。
4. 每日复盘格式。
5. 每周测评。
6. 阶段退出标准。
7. 弱点重新诊断机制。
8. 遇到计划偏差时如何调整。
9. CSP-S 一等奖目标下各阶段能力要求。

当前还缺一份：

> `60天CSP-S一等奖冲刺计划.md`

### P0：训练闭环还没真正建立

当前系统从历史数据出发生成了诊断和任务，但学生完成训练后，系统还不知道：

- 做了哪道题。
- 是否 AC。
- 提交了几次。
- 用了多久。
- 错因是什么。
- 是否复发旧弱点。
- 是否能写出部分分方案。
- 是否能独立复盘。

没有这些数据，就无法真正动态调整计划。

### P1：大模型生成题还没有质量验收

当前生成题包状态是：

```text
LLM_GENERATED_NEEDS_REVIEW
```

这很重要。它表示：

- 题面、题解、暴力、标程、生成器已经生成。
- 但还没有经过正式审题。
- 还没有编译验证。
- 还没有对拍。
- 还没有测试数据覆盖。
- 还没有人工确认无歧义。

所以这些题现在适合做“诊断题草稿”和“训练材料”，不适合作为正式比赛题。

### P1：分析报告还偏工程化，不够教练化

当前 `weakness_report.md` 更像系统输出，不像一位教练给学生看的训练诊断。

它需要进一步变成：

1. 学生当前水平判断。
2. 一等奖目标差距。
3. 最危险的失分场景。
4. 每个弱点对应的具体训练动作。
5. 每个动作如何验收。
6. 如果 7 天后没有改善，如何调整。

### P1：题目选择仍偏机械

当前选题逻辑主要基于历史证据题、模块、弱点标签。还不够考虑：

- 题目难度梯度。
- 题目之间的迁移关系。
- 一道题训练哪个具体动作。
- 是否适合当天状态。
- 是否适合作为阶段测评。
- 是否用于建立信心，还是用于暴露短板。

### P2：项目还缺少可交互产品界面

当前核心产物主要是 JSON 和 Markdown。

如果要给学生或家长使用，需要进一步有界面：

- 今日任务。
- 当前薄弱点。
- 题目详情。
- 复盘输入。
- 每周进步曲线。
- 60 天目标进度。
- 大模型复盘建议。

## 8. 建议下一步路线

### 第一步：把分析做准

建议先不要继续扩展出题数量，而是提高分析准确性。

具体动作：

1. 把 `DIAGNOSIS_LLM_LIMIT` 从 5 提高到 30。
2. 优先覆盖严重度最高、问题数最多、证据最多的问题簇。
3. 重新运行：

```powershell
$env:DIAGNOSIS_LLM_LIMIT='30'
$env:DIAGNOSIS_LLM_REQUIRE_REAL='true'
pnpm diagnosis:llm
pnpm diagnosis:propagate
pnpm diagnosis:report
pnpm problem-bank:all
```

目标产物：

- 更准确的 `weakness_report.md`
- 更可信的 `weakness_summary.json`
- 更贴近真实弱点的 `task_pack_7d.md`

验收标准：

- 至少前 30 个簇由 DeepSeek 真实诊断。
- top 10 weakness 变化可解释。
- 每个 top weakness 都能对应具体训练动作。

### 第二步：生成正式 60 天计划

基于更准的诊断，生成：

```text
docs/60天CSP-S一等奖冲刺训练计划.md
```

这份计划应该包含：

1. 总目标。
2. 阶段目标。
3. 每周主题。
4. 每日训练结构。
5. 每日验收指标。
6. 每周复盘模板。
7. 每 7 天重新诊断机制。
8. 风险预案。

### 第三步：建立训练日志

新增一个训练日志结构，例如：

```json
{
  "date": "2026-06-25",
  "problemId": "P8699",
  "weaknessId": "interval_dp_state_error_4",
  "taskType": "SELECT_EXISTING",
  "timeSpentMinutes": 95,
  "submissionCount": 3,
  "result": "AC",
  "scoreBeforeAc": [30, 70, 100],
  "preSolvePlanWritten": true,
  "partialScorePlan": {
    "score30": "暴力搜索或小 n DP",
    "score50": "简化约束",
    "score70": "优化状态或特殊性质",
    "score100": "完整模型"
  },
  "mistakes": [
    "len 枚举顺序错误",
    "dp[l][r] 初始化遗漏"
  ],
  "oldWeaknessReappeared": true,
  "reviewNote": "知道是区间 DP，但没有先写 len=1 初始化。"
}
```

没有训练日志，60 天计划就无法动态调整。

### 第四步：给生成题加质量验证

至少要做：

1. 编译 `bruteforce.cpp`
2. 编译 `std.cpp`
3. 编译 `generator.cpp`
4. 用 generator 生成小数据。
5. brute 和 std 对拍。
6. 输出对拍报告。
7. 人工确认题面和题解一致。

生成题通过这些验证前，都只能是：

```text
LLM_GENERATED_NEEDS_REVIEW
```

## 9. 60 天训练方案草案

下面是基于当前诊断的草案，不是最终版。最终版应在扩大 LLM 诊断覆盖后再生成。

### 阶段 1：第 1 到 7 天，验证诊断是否准确

目标：

- 不急着大量刷题。
- 验证 top 3 弱点是否真的命中。
- 建立每日复盘格式。

每天任务：

1. 1 道历史证据题。
2. 1 道同弱点变式题。
3. 每题必须写 30/50/70 分方案。
4. 每题必须记录提交次数和错因。

重点主题：

- 数学建模。
- 部分分拆解。
- 区间 DP。

退出标准：

- 能确认 top 3 weakness 是否真实命中。
- 能形成稳定复盘格式。
- 能判断后续 53 天是否需要调整优先级。

### 阶段 2：第 8 到 20 天，集中攻克三大主弱点

目标：

- 数学建模。
- 部分分策略。
- 区间 DP。

训练比例：

- 数学建模：30%
- 部分分拆解：30%
- 区间 DP：30%
- 复盘：10%

每日结构：

1. 20 分钟读题建模。
2. 60 到 90 分钟主训练题。
3. 30 到 40 分钟变式题或历史错题。
4. 30 分钟复盘。

退出标准：

- 每道难题能写出 30/50/70 分方案。
- 区间 DP 能稳定写出状态定义、初始化、枚举顺序。
- 同类题提交次数下降到 2 次以内。

### 阶段 3：第 21 到 35 天，扩展模型库

目标：

- 普通 DP。
- 图论建模。
- 二分答案。
- 并查集。
- 贪心。

训练方式：

- 每 3 天一个主题。
- 每个主题至少完成：
  - 1 道基础题。
  - 1 道中档题。
  - 1 道难题部分分拆解。
  - 1 道变式题。

退出标准：

- 能快速识别题目属于哪个模型。
- 能说明 check 单调性。
- 能说明图论建模中的点、边、状态、约束。
- 并查集题能明确集合含义和合并条件。

### 阶段 4：第 36 到 50 天，模拟赛化训练

目标：

- 从专题能力转成考场能力。

每周安排：

- 2 次完整模拟。
- 2 次 T3/T4 专项训练。
- 1 次错题重做。
- 1 次弱点复诊。

重点观察：

- T1/T2 是否稳定拿分。
- T3 是否能拿 30/50/70。
- T4 是否能快速判断可拿部分分。
- 是否还会在区间 DP、图论、二分上犯旧错。

退出标准：

- 模拟赛中能稳定保住基础分。
- 难题能写出部分分方案。
- 不因边界和初始化丢大量分。

### 阶段 5：第 51 到 60 天，查漏补缺和考试策略

目标：

- 稳定性。
- 时间分配。
- 题目取舍。
- 心态和复盘。

每天重点：

1. 一道中档题保持手感。
2. 一道难题只做分析和部分分。
3. 一次错因复盘。
4. 一次考场策略训练。

最终验收：

- 能把 T3/T4 拆成可执行部分分。
- 能把高风险题提前识别出来。
- 能把提交次数和调试时间压下来。
- 能在模拟赛里减少低级失误。
