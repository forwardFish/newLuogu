# AI 信奥训练教练项目完整总结文档

生成日期：2026-06-25  
适用项目：`AI 信奥训练教练 / CSP-S 目标分闭环训练系统 / newLuogu`  
用途：给 Codex、合作老师、产品合伙人、商业化验证、后续开发排期使用  
版本：v1.0 综合理解版

## 0. 最重要的结论

这个项目不是普通刷题软件，也不是题解网站，也不是训练营包装。

它真正要做的是：

> **用 AI 替代信奥老师的大部分日常训练决策工作：诊断学生、判断短板、安排下一题、监督训练、分层提示、错题复盘、动态调整、模拟赛验证，最后在必要时进行题目讲解。**

项目的核心不是“让 AI 会讲题”，而是让 AI 像一个长期陪跑的竞赛教练一样，持续回答：

```text
1. 这个学生当前到底在哪里？
2. 他的目标是 CSP-S 一等奖还是明确目标分？
3. 他离目标差在哪里？
4. 哪个短板最可能造成考场失分？
5. 今天应该练哪道题？
6. 为什么是这道题？
7. 做完以后暴露了什么？
8. 系统之前的诊断是否命中？
9. 明天或下周计划是否需要调整？
10. 是否真的更接近目标分或一等奖稳定区间？
```

讲解不是入口，而是最后一环。  
如果一上来就讲题，系统会退化成题解网站。真正的 AI 老师应该先诊断、先提示、先引导学生思考，只有在学生卡住、反复提交失败或复盘阶段，才进入完整讲解。

## 1. 项目最终定位

### 1.1 一句话定义

```text
AI 信奥训练教练是一套基于洛谷历史提交、代码行为、CSP-S 真题基准、目标分模型和训练反馈的 AI 教练系统。
它通过“诊断 → 出题 → 训练 → 记录 → 复盘 → 调整 → 模拟赛验证 → 重新诊断”的闭环，
帮助学生从随机刷题升级为目标分驱动的高效训练。
```

### 1.2 不是什么

本系统不是：

```text
1. 不是普通题单生成器
2. 不是随机刷题推荐器
3. 不是只会输出报告的分析工具
4. 不是题解搬运网站
5. 不是简单的 AI 视频讲解库
6. 不是保证一等奖的承诺工具
7. 不是老师讲课视频的替代剪辑系统
8. 不是直接让大模型读完全部代码再给结论
```

### 1.3 是什么

本系统是：

```text
1. 目标分驱动的刷题训练引擎
2. 基于历史提交和代码行为的诊断系统
3. 基于 CSP-S 真题与知识能力树的差距分析系统
4. 以训练反馈为核心的 AI 教练闭环系统
5. 分层提示、错因讲解、复盘和过关判定系统
6. 可以商业化为“AI 信奥教练”的长期陪跑产品
```

## 2. 项目的核心目标

### 2.1 从“刷更多题”变成“刷最该刷的题”

传统刷题方式的问题是：

```text
题很多，但不知道为什么刷
刷完了，但不知道是否真的提升
错了，但不知道是读题、建模、状态、实现还是策略问题
看了题解，但下次还是不会迁移
做了很多题，但模拟赛仍然不稳定
```

本项目的目标是把刷题变成：

```text
目标分
  ↓
能力差距
  ↓
预计失分
  ↓
训练优先级
  ↓
今日任务
  ↓
做题记录
  ↓
错因诊断
  ↓
复盘动作
  ↓
知识点过关
  ↓
套题验证
```

### 2.2 支持两类目标

系统可以服务两类目标：

```text
目标 A：CSP-S 一等奖
目标 B：明确目标分，例如 200 / 240 / 260 / 280
```

如果目标是一等奖，系统必须结合：

```text
1. 目标省份
2. 目标年份
3. 近几年一等奖线
4. 安全分数垫
5. 真实模拟赛成绩
6. T1/T2/T3/T4 稳定性指标
```

如果目标是明确分数，例如 200 分，系统必须拆成：

```text
T1：80-100 分，必须稳定拿分
T2：60-90 分，主要提分区
T3：30-60 分，训练部分分
T4：0-30 分，策略拿分区
```

## 3. 为什么这个项目不是“训练营”，而是“AI 老师替代系统”

### 3.1 训练营只是商业包装，不是项目本质

30 天训练闭环服务可以作为商业化交付形态，但项目的核心不是训练营。

训练营通常是：

```text
固定课程
固定题单
固定节奏
老师统一讲
学生被动完成
```

而这个项目应该是：

```text
每个学生一套诊断
每个短板一组证据
每道题有训练目的
每次失败回填系统
每周动态调整计划
每个知识点有过关标准
每次讲解都服务于真实错因
```

### 3.2 老师真正做的事情

一个优秀信奥老师的核心价值不是只会讲题，而是：

| 老师工作 | AI 替代方式 |
|---|---|
| 判断学生当前水平 | 洛谷提交 + 代码分析 + 真题记录 + 模拟赛 |
| 找出真实短板 | 题目级诊断 + 问题簇 + 版本差异 + 错误类型 |
| 判断下一题练什么 | 目标分差距 + 预计失分 + 知识点掌握度 |
| 判断学生为什么错 | 结果轨迹 + 代码风险 + 做题阶段日志 |
| 给提示而不是直接给答案 | Hint Engine 分层提示 |
| 做错题复盘 | Review Engine |
| 安排重做和变式题 | Problem Ladder + Mutate |
| 动态调整计划 | Coach State + Adjust Engine |
| 做阶段验证 | Mock Exam Engine |
| 必要时讲解 | Explanation Engine |

所以系统的产品定位应当是：

```text
AI 信奥教练 = 诊断系统 + 出题系统 + 训练监督系统 + 提示系统 + 复盘系统 + 调整系统 + 讲解系统
```

讲解只是最后一环。

## 5. 当前学生数据画像

根据已分析的历史代码与提交记录，当前已有数据基础大致如下：

```text
代码文件总数：2345
题目总数：829
已 AC 题目：614
未 AC / 未知题目：215
平均每题提交次数：2.83
静态风险文件数：1762
语言主要为 C++，另有少量 Text/Other 和 Python
```

当前四维训练诊断分：

```text
T1 稳定拿分：75.51
T2 建模转化：81.48
T3 部分分策略：66.46
T4 综合策略：71.53
综合分：73.75
```

这些分数不是比赛预测分，而是训练诊断分。  
它们用于判断后续训练优先级，而不能直接宣称学生真实 CSP-S 分数。

### 5.1 当前画像解读

```text
1. 题型覆盖较广，不是完全缺算法积累。
2. T2 建模转化相对较强，说明中档题经验不差。
3. T1 稳定性仍有实现风险，尤其是边界、数组、long long、初始化。
4. T3 部分分策略是明显短板，需要系统训练。
5. T4 综合策略需要通过真实模拟赛验证。
6. 平均提交次数偏高，说明调试成本和实现稳定性仍需改善。
7. 高提交次数旧题是非常有价值的复盘素材。
```

### 5.2 当前不能过度相信的结论

当前系统已经生成过部分弱点和模拟结果，但必须区分：

```text
真实证据
  历史提交、代码文件、分数轨迹、代码静态特征、真实 LLM 调用 traces

模拟证据
  30 天模拟训练、模拟 daily attempts、模拟 mock exam scores、readiness 曲线
```

例如：

```text
312 分、readiness 0.87 是模拟结果，不是学生真实套题成绩。
不能拿来证明已经接近一等奖。
```

## 8. 第二层：题目识别与代码分析层

### 8.1 核心原则

先结构化数据，不要让大模型直接读取全部源码。

因为直接把 829 道题、2345 份代码交给大模型会导致：

```text
1. 成本高
2. 上下文过长
3. 相似问题重复分析
4. 输出空泛
5. 无法稳定形成结构化诊断
```

正确流程是：

```text
本地代码目录
  ↓
submission_manifest.json
  ↓
code_static_metrics.json
  ↓
problem_code_timeline.json
  ↓
algorithm_module_stats.json
  ↓
error_pattern_candidates.json
  ↓
deep_review_samples.json
  ↓
csp_s_risk_profile.json
```

### 8.2 每道题必须聚合

不能只看单份代码文件。  
必须按 `problemPid` 聚合所有提交版本：

```json
{
  "problemPid": "P8699",
  "attemptCount": 16,
  "hasAc": true,
  "bestScore": 100,
  "scoreTrajectory": [0, 40, 40, 50, 100],
  "scorePattern": "PARTIAL_TO_AC",
  "resultPattern": "MULTI_FAIL_THEN_AC"
}
```

### 8.3 代码分析目标

对每份代码提取：

```text
1. 行数
2. main 函数长度
3. 函数数量
4. 循环数量
5. 最大嵌套深度
6. 是否递归
7. 是否全局数组
8. 是否 #define int long long
9. 是否使用 memset
10. 是否有数组边界风险
11. 是否有整型溢出风险
12. 是否长代码
13. 是否模板依赖明显
```

## 9. 第三层：题目级诊断层

### 9.1 为什么需要题目级诊断

粗粒度统计只能告诉你：

```text
DP 平均提交次数高
并查集平均提交次数高
部分分题多
数组风险多
```

但真正有训练价值的问题是：

```text
这道题到底哪里错了？
是读题错？
是模型错？
是状态定义错？
是转移错？
是初始化错？
是边界错？
是复杂度错？
是没有想到部分分？
是调试超时？
```

所以每道题都必须输出诊断对象。

### 9.2 题目诊断结构

每道题建议输出：

```json
{
  "problemPid": "Pxxxx",
  "attemptCount": 0,
  "solved": true,
  "bestScore": 100,
  "scorePattern": "PARTIAL_TO_AC",
  "modules": ["dp"],
  "diagnosisLevel": "HIGH_VALUE_REVIEW",
  "mainWeakness": "DP 状态设计和边界处理不稳定",
  "likelyErrorTypes": [
    "STATE_ERROR",
    "BOUNDARY_ERROR",
    "INIT_ERROR"
  ],
  "evidence": [
    "提交次数高于平均值",
    "出现多次部分分后才 AC",
    "代码风险包含数组边界",
    "版本变化中增加了初始化逻辑"
  ],
  "versionInsights": {},
  "trainingAction": {
    "recommendation": "安排同类题训练",
    "verifyMethod": "提交次数 <= 2，能解释状态和初始化"
  }
}
```

### 9.3 错误类型枚举

```text
READING_ERROR
MODEL_ERROR
ALGO_ERROR
STATE_ERROR
TRANSITION_ERROR
BOUNDARY_ERROR
INDEX_ERROR
OVERFLOW_ERROR
INIT_ERROR
MULTITEST_ERROR
COMPLEXITY_ERROR
DEBUG_TIMEOUT
STRATEGY_ERROR
PARTIAL_SCORE_MISSED
TEMPLATE_UNFAMILIAR
IMPLEMENTATION_RISK
UNKNOWN
```

### 9.4 题目诊断等级

```text
NORMAL
WATCH
HIGH_VALUE_REVIEW
CRITICAL_WEAKNESS
```

高提交、多次部分分、长期未 AC、最终 AC 但过程曲折的题，应该进入 `HIGH_VALUE_REVIEW` 或 `CRITICAL_WEAKNESS`。

## 10. 第四层：问题簇与 LLM 诊断层

### 10.1 目的

不是让大模型逐题读完所有代码，而是：

```text
全题目规则诊断
  ↓
问题簇归类
  ↓
代表题选择
  ↓
构建 evidence pack
  ↓
大模型分析代表题共性
  ↓
诊断回填到同簇所有题
```

### 10.2 问题簇构成

聚类不应只看算法模块，而应综合：

```text
1. primaryModule
2. scorePattern
3. resultPattern
4. riskTags
5. candidateErrorTypes
6. cspAbilityRole
7. difficultyBand
8. versionChangePattern
```

### 10.3 代表题选择

每个簇选 3-5 道代表题：

```text
1. 最典型的一道
2. 最严重的一道
3. 分数轨迹最有信息量的一道
4. 最近做的一道
5. 版本变化明显的一道
```

### 10.4 LLM 只做什么

大模型只做：

```text
1. 分析代表题之间的共性问题
2. 总结问题簇主薄弱点
3. 判断错误类型
4. 生成训练动作说明
5. 提醒不确定性
```

大模型不能做：

```text
1. 凭空下结论
2. 把历史 AC 当作考场稳定
3. 没有盲测证据就确认短板
4. 直接决定一等奖 readiness
5. 生成未经校验的正式题目
```

## 11. 第五层：CSP-S 基准库层

### 11.1 为什么需要基准库

如果没有 CSP-S 基准库，系统只能回答：

```text
你历史代码哪里异常
```

但不能回答：

```text
这些异常对 CSP-S 一等奖到底有多重要
它会影响 T1、T2、T3 还是 T4
如果不修，可能丢多少分
下一轮应优先验证哪个短板
```

### 11.2 必须建立两个核心库

#### A. 知识能力树

文件：

```text
data/csp-s-benchmark/knowledge_ability_tree.json
```

它不是简单复制 NOI 大纲，而是拆成：

```text
一级领域
二级模块
三级可观察能力
四级错误模式
CSP-S 题位相关性
训练方式
验证方式
```

示例：

```text
DP
  DP.INTERVAL
    DP.INTERVAL.RECOGNIZE_STRUCTURE
    DP.INTERVAL.INIT_AND_ORDER
```

#### B. CSP-S 历年真题库

文件：

```text
data/csp-s-benchmark/past_problems_2019_2025.json
```

必须覆盖：

```text
2019-2025
每年 T1/T2/T3/T4
共 28 道全国统一 CSP-S 第二轮提高组真题
```

每题字段：

```text
年份
题位
题号
题名
官方来源
洛谷链接
主知识点
次知识点
三级能力点
非知识点能力
建模类型
关键转化
实现风险
部分分结构
一等目标分
常见失败模式
分类置信度
```

### 11.3 需要人工确认的内容

以下内容不能完全自动化：

```text
1. 每年 T1/T2/T3/T4 顺序
2. 每道真题主知识点
3. 部分分结构
4. 一等奖目标选手合理目标分
5. 某题是否属于全国统一题
6. 学生是否看过题解
7. 学生是否盲测完成
```

## 12. 第六层：目标分与预计失分层

### 12.1 核心思想

训练优先级不应只由“历史错题多不多”决定，而应由：

```text
预计能减少多少 CSP-S 考场失分
```

决定。

### 12.2 证据权重

建议证据权重：

```text
CSP-S 真题盲测独立完成：1.00
四题模拟赛中完成：0.95
未做过同类新题独立完成：0.80
历史题一次 AC：0.65
历史题多次提交后 AC：0.35
看过题解后 AC：0.15
规则推断但未验证：0.10
```

### 12.3 掌握等级

```text
NO_EVIDENCE
EXPOSED
UNSTABLE
BASIC
TRANSFERABLE
CONTEST_STABLE
```

### 12.4 预计失分公式

```text
expectedLoss =
  cspFrequency
× slotImpact
× scoreWeight
× failureProbability
× unrecoveredScoreRatio
× firstPrizeSensitivity
× confidence
```

这不是伪精确预测，而是训练优先级排序工具。

### 12.5 输出结果

```text
expected_score_loss.json
first_prize_gap_report.md
first_prize_gap_report.json
```

报告必须回答：

```text
1. 当前目标分是多少
2. 高风险预计失分有多少
3. 需要压降多少风险
4. 哪些能力点最值得优先训练
5. 哪些结论只是待验证假设
6. 下一轮盲测题是什么
```

## 13. 第七层：出题与训练任务层

### 13.1 出题原则

出题优先级：

```text
Select > Mutate > Generate
```

解释：

```text
Select：优先从成熟题库和洛谷题中选题
Mutate：基于薄弱点生成同构变式题
Generate：少量生成原创诊断题
```

默认比例：

```text
Select：60%-70%
Mutate：20%
Generate：10%-20%
```

### 13.2 Select

用于：

```text
1. T1 稳定性训练
2. T2 模块训练
3. 标准算法训练
4. 需要成熟评测的训练
```

输出：

```text
题号
题目来源
训练目标
为什么选
对应能力点
限时
验收标准
```

### 13.3 Mutate

用于：

```text
1. 验证迁移能力
2. 防止学生只是记住原题
3. 改变约束、输出目标、特殊性质
```

注意：

```text
不能只是换故事背景
不能复制洛谷题面
不能小明换小红
必须改变真正的算法或模型验证点
```

### 13.4 Generate

用于：

```text
1. 专项诊断
2. 部分分训练
3. 模拟赛 T3/T4 子任务训练
4. 迁移能力检查
```

原创题必须有：

```text
题面
输入输出
数据范围
样例
标准解
暴力解
数据生成器
对拍器设计
测试点设计
难度定位
对应薄弱点
验收标准
```

未经过编译、对拍、审题的生成题，只能标记为：

```text
LLM_GENERATED_NEEDS_REVIEW
DRAFT_NOT_FOR_FORMAL_TEST
```

不能作为正式训练题。

## 15. 第九层：提示与讲解系统

### 15.1 讲解是最后一环

AI 老师不能一开始就给完整题解。

正确顺序是：

```text
做题前问题
  ↓
学生尝试
  ↓
卡住时分级提示
  ↓
提交后错因讲解
  ↓
复盘讲解
  ↓
必要时完整题解
  ↓
同构题重做
```

### 15.2 分层提示等级

```text
Hint 1：方向提示
Hint 2：模型提示
Hint 3：关键步骤
Hint 4：解法框架
Hint 5：完整题解
```

提示使用也要进入能力评价：

```text
完全独立 AC > Hint 1 AC > Hint 3 AC > 看完整题解 AC
```

### 15.3 提交后讲解

提交后讲解应针对学生错误，而不是泛泛讲题：

```text
这题你错在 check 函数含义写反
这题你错在 dp 初始化没有覆盖 len=1
这题你错在没有先写 30 分暴力
这题你错在调试时间过长，没有及时止损
```

### 15.4 视频素材使用原则

网上视频可以作为知识素材库，但不建议直接下载、改字幕、换声音后商业化。

正确方式：

```text
公开视频 / 授权视频
  ↓
转写
  ↓
切片
  ↓
知识点标签
  ↓
内部检索
  ↓
生成原创讲解
  ↓
推荐原视频片段或输出自有图文/音频讲解
```

视频最终不是课程库，而是：

```text
知识点片段库
常见错因片段库
题型讲解素材库
```

## 17. 教练状态与动态调整

### 17.1 诊断必须是“假设”，不是最终结论

每个 weakness 应有：

```text
confidence
priority
hitCount
missCount
lastValidatedAt
status
requiredValidationTasks
```

状态：

```text
HYPOTHESIS
VALIDATED
WATCH
DOWNGRADED
REJECTED
```

### 17.2 命中标准

```text
命中率 ≥ 70%：确认薄弱点
50%-70%：中等可信
30%-50%：弱假设
<30%：大概率误判
未盲测：不允许下最终结论
```

### 17.3 训练后纠偏

如果预测失败且实际失败：

```text
提高置信度
继续安排同类但不同语境题
```

如果预测失败但实际通过：

```text
降低优先级
检查历史分类是否误判
```

如果出现新问题：

```text
新增 weakness candidate
加入下周验证任务
```

如果连续多道盲测通过：

```text
能力升级为 TRANSFERABLE 或 CONTEST_STABLE
训练比例下降
```

## 18. 模拟赛与达标判断

### 18.1 不能用刷题数量判断

进入一等奖稳定区间不能看刷了多少题。必须看：

```text
T1/T2 正确率
T1/T2 用时
T3 部分分
T4 子任务识别
四题模拟赛总分
连续模拟赛稳定性
低级错误数量
一次 AC 率
平均提交次数
```

### 18.2 建议达标线

```text
T1/T2 正确率 >= 90%
T1 平均用时 <= 35 分钟
T2 平均用时 <= 60 分钟
T3 至少稳定拿 30 到 60 分
T4 至少能识别可拿分子任务
四题模拟赛总分 >= 目标省份一等奖线 + 20 分安全垫
连续 5 场模拟赛达到目标
平均提交次数下降
一次 AC 率提高
低级错误数量下降
```

### 18.3 readiness 状态

```text
NOT_READY
APPROACHING
STABLE_ZONE
HIGH_CONFIDENCE
```

只有真实模拟赛和真实训练日志达标，才能进入 `STABLE_ZONE`。  
模拟数据只能说明系统逻辑可运行，不能证明学生真实达标。

## 21. 合规与边界

### 21.1 数据边界

```text
只使用公开数据、用户授权数据、手动导入数据。
不保存账号密码。
不绕过平台限制。
不自动提交代码。
不抓取非公开数据。
```

### 21.2 未成年人数据

如果用于中小学生，必须考虑监护人授权、最小必要收集、数据删除、数据导出和隐私说明。

### 21.3 内容版权

网上讲解视频不能直接搬运、改配音、改字幕后商业化。

可接受方式：

```text
1. 保存公开链接
2. 推荐原视频片段
3. 对授权视频做切片
4. 抽取知识点后生成原创讲解
5. 使用自己生成的图文、音频和视频讲解
```

### 21.4 一等奖承诺边界

不能承诺：

```text
刷完 30 天一定一等奖
系统判断你一定能拿一等奖
模拟数据证明你已经达标
```

可以承诺：

```text
训练目标明确
每日任务明确
每题目的明确
训练结果可记录
每周可复盘
未达标可调整
达标可验证
模拟赛可追踪
```

## 25. 最终产品愿景

最终产品不是“AI 题解助手”，而是：

```text
AI 信奥教练系统
```

它的长期形态是：

```text
学生每天打开系统
  ↓
系统告诉他今天练什么
  ↓
系统解释为什么练
  ↓
学生做题
  ↓
卡住时系统分层提示
  ↓
提交失败后系统判断错因
  ↓
做完后系统要求复盘
  ↓
系统判断该知识点是否过关
  ↓
系统调整下一题
  ↓
每周做套题验证
  ↓
每月给家长和老师报告
```

系统的壁垒不在“有多少题解”，而在：

```text
1. 能否持续跟踪学生真实能力
2. 能否准确判断错因
3. 能否把错因转成下一题
4. 能否根据训练反馈纠偏
5. 能否最终证明训练对目标分有帮助
```

## 26. 当前最关键的下一步

建议下一步不要继续泛泛扩展功能，而是按顺序完成：

```text
1. 确认 CSP-S 真题库和知识能力树
2. 完善题目级诊断和模块误判校准

7. 让每日任务由最高预计失分能力点驱动
8. 建立分层提示和错因讲解
9. 每周用模拟赛和复盘调整计划
10. 再考虑商业化页面和支付
```

---

## 27. 结尾总结

这个项目的本质不是“做一个更聪明的刷题网站”，而是：

> **做一个围绕 CSP-S 目标分的 AI 竞赛教练。**

它从学生历史数据出发，但不止于历史数据；  
它使用大模型，但不让大模型凭空决策；  
它会生成题，但不依赖原创题证明价值；  
它有讲解，但讲解不是入口；  
它能商业化，但第一性原理是训练有效，而不是包装训练营。

最终系统要做到：

```text
没有题目级证据，不下结论。
没有真题盲测，不确认短板。
没有目标分映射，不安排训练优先级。
没有训练反馈，不宣布提升。
没有连续稳定表现，不说接近一等奖。
```

只有这样，它才不是普通 AI 工具，而是一个真正有机会替代部分信奥老师日常训练工作的 AI 教练系统。
