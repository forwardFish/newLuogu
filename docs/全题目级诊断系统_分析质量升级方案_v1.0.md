# 全题目级诊断系统：分析质量升级方案 v1.0

生成日期：2026-06-24

本文档用于说明当前代码分析系统为什么“不够找到问题”，以及下一阶段如何实现“全题目级诊断系统”。目标是从粗粒度统计画像升级为逐题诊断、模块弱点定位、错误类型推断和针对性训练计划生成。

## 1. 背景与问题

当前系统已经能读取本地洛谷代码文件，并生成以下分析结果：

```text
submission_manifest.json
code_static_metrics.json
problem_code_timeline.json
algorithm_module_stats.json
error_pattern_candidates.json
deep_review_samples.json
csp_s_risk_profile.json
final_code_analysis_report.md
```

当前分析能回答：

```text
1. 一共有多少代码文件
2. 做过多少题
3. AC 多少题
4. 哪些题提交次数多
5. 哪些代码有静态风险
6. 大致覆盖哪些算法模块
7. T1/T2/T3/T4 当前分数是多少
8. 下一步训练比例大致如何
```

但是用户反馈是正确的：**当前分析质量还不够。**

原因是当前分析更偏：

```text
统计画像
模块计数
静态风险扫描
训练比例估算
```

而不是：

```text
逐题诊断
错因定位
薄弱点归因
版本差异分析
证据链分析
针对性出题
```

所以当前系统会说：

```text
并查集平均提交次数偏高
区间 DP 平均提交次数偏高
数组风险较多
部分分题较多
疑似卡题 86 道
```

但它还不能稳定回答：

```text
这道题到底哪里错了？
是状态设计错？
是转移顺序错？
是边界错？
是复杂度错？
是读题错？
是没有想到部分分？
是调试策略错？
是模板不熟？
```

这就是下一阶段要解决的问题。

## 2. 下一阶段目标

下一阶段要实现：

```text
全题目级诊断系统
```

核心目标：

```text
对所有已做过的题目逐题生成诊断结果，
再从逐题诊断中汇总出真正的薄弱点，
最后把薄弱点转化为后续训练动作。
```

目标不是简单输出：

```text
你要多练 DP
你要多练图论
你要注意边界
```

而是输出：

```text
你在区间 DP 上的问题主要是枚举顺序和初始化。
证据来自 P8699、P1435、U467266 等题。
这些题共同表现为多次部分分、提交次数高、代码反复修改 DP 数组与边界。
下一步应训练 4 道区间 DP 题，每题必须写 len 枚举、状态定义、初始化和边界。
```

## 3. 当前分析方案回顾

当前分析主要包含以下步骤。

### 3.1 文件级分析

读取本地代码文件：

```text
data/code/.../codes/
```

生成：

```text
submission_manifest.json
```

每份代码记录：

```text
文件路径
题号
提交记录 ID
结果
分数
语言
提交时间
文件大小
```

### 3.2 静态代码分析

生成：

```text
code_static_metrics.json
```

每份代码分析：

```text
代码行数
非空行数
最大行长
函数数量
循环数量
最大嵌套深度
是否使用全局数组
是否使用 #define int long long
是否递归
是否疑似数组边界风险
是否疑似整型溢出风险
是否长代码
```

### 3.3 题目时间线分析

生成：

```text
problem_code_timeline.json
```

按题目聚合：

```text
提交次数
是否 AC
最高分
首次提交时间
最后提交时间
是否多次失败后 AC
是否长期未 AC
是否部分分
是否长代码题
```

### 3.4 算法模块识别

生成：

```text
algorithm_module_stats.json
```

通过源码特征粗略识别：

```text
DP
图论
二分
并查集
区间 DP
搜索
字符串
数学
线段树
树状数组
```

### 3.5 当前不足

当前不足在于：

```text
1. 模块识别是关键词级，不够精确
2. 没有逐题诊断主结论
3. 没有比较同一题多个版本之间的差异
4. 没有从分数变化推断错误阶段
5. 没有把错因拆成明确错误类型
6. 没有把每个薄弱点绑定证据题目
7. 没有把每个薄弱点转成具体训练动作
```

## 6. 每题诊断字段说明

### 6.1 diagnosisLevel

用于判断这道题是否值得深度复盘。

枚举：

```text
NORMAL
WATCH
HIGH_VALUE_REVIEW
CRITICAL_WEAKNESS
```

规则：

```text
NORMAL：
一次 AC 或提交较少，风险不高

WATCH：
提交次数略高，或存在明显静态风险

HIGH_VALUE_REVIEW：
提交次数高、多次部分分、最终 AC、长代码、模块薄弱

CRITICAL_WEAKNESS：
长期未 AC、高提交次数、部分分停滞、模拟 CSP-S 关键模块
```

### 6.2 likelyErrorTypes

错误类型枚举：

```text
READING_ERROR
MODEL_ERROR
ALGO_ERROR
STATE_ERROR
TRANSITION_ERROR
BOUNDARY_ERROR
INDEX_ERROR
OVERFLOW_ERROR
INIT_ERROR
MULTITEST_ERROR
COMPLEXITY_ERROR
DEBUG_TIMEOUT
STRATEGY_ERROR
PARTIAL_SCORE_MISSED
TEMPLATE_UNFAMILIAR
IMPLEMENTATION_RISK
UNKNOWN
```

注意：这些是系统根据提交轨迹、代码特征和版本变化推断的候选错误类型，不等同于人工最终结论。

### 6.3 mainWeakness

每题必须给出一句主诊断。

不能写：

```text
需要多练
有问题
不稳定
```

应该写：

```text
区间 DP 的状态设计和初始化不稳定
并查集集合含义和离散化细节不稳定
二分答案 check 单调性和边界需要加强
长代码调试成本高，容易隐藏边界错误
难题部分分拆解意识不足
```

## 7. 题目级诊断规则

### 7.1 提交次数规则

```text
attemptCount = 1 且 AC：
  可能是稳定题，诊断级别 NORMAL

attemptCount 2-4 且 AC：
  轻微调试，诊断级别 WATCH

attemptCount >= 5 且 AC：
  多次失败后 AC，诊断级别 HIGH_VALUE_REVIEW

attemptCount >= 8 且未 AC：
  诊断级别 CRITICAL_WEAKNESS
```

### 7.2 分数轨迹规则

根据每题提交分数变化判断问题。

```text
长期 0 分：
  可能是模型错误、算法错误、读题错误或复杂度错误

从 0 到部分分：
  可能逐步修正了暴力或基础情况

长期停留在部分分：
  可能不会正解，或缺少关键优化

部分分多次波动：
  可能边界、初始化、特殊情况处理不稳

最后 AC：
  可以比较早期版本与最终版本，提取修正方向
```

### 7.3 静态风险规则

风险标签到错误类型映射：

```text
ARRAY_INDEX_RISK -> INDEX_ERROR / BOUNDARY_ERROR
LARGE_GLOBAL_ARRAY -> INDEX_ERROR / MEMORY_RISK
DEFINE_INT_LONG_LONG -> OVERFLOW_ERROR / TEMPLATE_RISK
RECURSION -> BOUNDARY_ERROR / STACK_RISK
POSSIBLE_INT_OVERFLOW -> OVERFLOW_ERROR
MEMSET_SENTINEL -> INIT_ERROR
LONG_CODE -> DEBUG_TIMEOUT / IMPLEMENTATION_RISK
DEEP_LOOP_NESTING -> COMPLEXITY_ERROR / IMPLEMENTATION_RISK
DEEP_BRACE_NESTING -> IMPLEMENTATION_RISK
PARTIAL_SCORE_CODE -> PARTIAL_SCORE_MISSED
```

### 7.4 模块规则

不同模块对应不同典型错因。

```text
interval_dp:
  STATE_ERROR
  TRANSITION_ERROR
  INIT_ERROR
  BOUNDARY_ERROR

dp:
  STATE_ERROR
  TRANSITION_ERROR
  INIT_ERROR

dsu:
  MODEL_ERROR
  TEMPLATE_UNFAMILIAR
  INDEX_ERROR

binary_search:
  MODEL_ERROR
  BOUNDARY_ERROR
  COMPLEXITY_ERROR

graph:
  MODEL_ERROR
  COMPLEXITY_ERROR
  INDEX_ERROR

shortest_path:
  INIT_ERROR
  COMPLEXITY_ERROR
  TEMPLATE_UNFAMILIAR

segment_tree:
  TEMPLATE_UNFAMILIAR

fenwick_tree:
  INDEX_ERROR
```

## 8. 版本对比分析

同一道题通常有多个提交版本。

应该选出：

```text
firstVersion：第一次提交
bestVersion：最高分版本
finalVersion：最后一次提交
firstAcVersion：第一次 AC 版本
```

比较内容：

```text
代码行数变化
风险标签变化
模块特征变化
分数变化
是否从部分分到 AC
是否从短代码变成长代码
是否增加了数据结构
是否增加了 DP 数组
是否增加了 long long
是否增加了初始化逻辑
```

输出：

```json
{
  "problemPid": "Pxxxx",
  "versionCount": 8,
  "scorePattern": "0 -> 30 -> 70 -> 100",
  "lineCountChange": "+85",
  "riskFlagChange": {
    "added": ["ARRAY_INDEX_RISK"],
    "removed": ["POSSIBLE_INT_OVERFLOW"]
  },
  "likelyChangeReason": "从暴力逐步改为 DP 正解，最终修复边界和初始化。"
}
```

## 9. 模块弱点矩阵

输出：

```text
module_weakness_matrix.json
```

每个模块计算：

```json
{
  "module": "interval_dp",
  "problemCount": 79,
  "acProblemCount": 60,
  "unsolvedCount": 19,
  "avgAttempts": 4.19,
  "highAttemptProblemCount": 18,
  "partialScoreProblemCount": 10,
  "criticalProblemCount": 5,
  "riskProblemCount": 30,
  "weaknessScore": 86.5,
  "mainWeakness": "区间 DP 状态设计、初始化和枚举顺序不稳定",
  "evidenceProblems": ["P8699", "P1435", "U467266"]
}
```

### 9.1 weaknessScore 计算建议

```text
weaknessScore =
  avgAttempts * 12
  + unsolvedRatio * 25
  + partialScoreRatio * 20
  + highAttemptRatio * 20
  + riskProblemRatio * 15
  + criticalProblemRatio * 25
```

归一化到 0-100。

### 9.2 模块薄弱点排序

输出 Top 10：

```text
1. interval_dp
2. dsu
3. binary_search
4. state_compression_dp
5. segment_tree
6. graph
7. dp
...
```

排序依据不是简单题量，而是：

```text
提交次数
未 AC
部分分
高风险代码
关键 CSP-S 相关度
```

## 12. 针对性训练计划

输出：

```text
targeted_training_plan.md
```

不是普通 7 天计划，而是基于诊断结果的“薄弱点 → 训练动作”计划。

示例：

```md
## 薄弱点 1：区间 DP 状态设计和初始化

证据：
- P8699：16 次提交，多次部分分后 AC
- U467266：16 次提交
- P1435：高提交次数，区间 DP 特征明显

训练动作：
1. 练 2 道基础区间 DP
2. 练 2 道中档区间 DP
3. 每题必须写状态、转移、初始化、len 枚举顺序

验收：
- 同类题提交次数 <= 2
- 能口头解释为什么按 len 枚举
- 能写出边界初始化
```

## 15. 验收标准

必须满足：

```text
1. 对所有题目生成诊断，不只挑样本
2. 每题有主诊断 mainWeakness
3. 每题有错误类型 likelyErrorTypes
4. 每题有 evidence
5. 每题有 trainingAction
6. 输出模块弱点矩阵
7. 输出错误类型分布
8. 输出版本差异分析
9. 输出 weakness_report.md
10. 输出 targeted_training_plan.md
```

质量验收：

```text
1. 不再只说“多练 DP”
2. 必须说明“为什么是这个弱点”
3. 必须列出证据题目
4. 必须区分不同 DP 问题，例如状态设计、转移、初始化、枚举顺序
5. 必须区分并查集是集合含义、离散化、带权并查集还是模板不熟
6. 必须区分二分是 check 单调性、边界还是答案定义
7. 必须识别部分分策略问题
8. 必须能指导下一步具体选题
```

## 16. 不合格情况

以下情况视为不合格：

```text
1. 只输出总体统计
2. 只输出模块计数
3. 只输出训练计划，不输出诊断证据
4. 只分析高提交题，不覆盖所有题目
5. 每题没有 mainWeakness
6. 没有错误类型推断
7. 没有训练动作
8. 不能解释为什么某个模块是薄弱点
9. 不能从旧题映射到后续训练
```

## 17. 最终目标
